Notion de test

Durée :

Environnement de travail :

Pré-requis : Python3

Contexte :

Écrire le code de test en parallèle du code à exécuter est une bonne habitude. Utilisés à bon escient, les tests permettent de définir avec plus de précision l'intention de votre code et d'avoir une architecture propre. Vous allez apprendre à différencier à rédiger vos premiers tests avec et sans les bibliothèques proposées par Python3.

Objectifs

  • Comprendre l'utilité des tests

  • Différencier les tests manuels, automatiques, unitaires, d'intégration et fonctionnels

  • Utiliser les assertions

Contexte :

Le véritable gain est dans le futur : quand vous modifiez votre code, vous pourrez rapidement déterminer s'il est cassé. Plus vous avancez sur des projets complexes, plus votre code va s'épaissir. Vous ne vous souviendrez pas de toutes les dépendances, des effets de bords et des interactions.

Différencier les tests automatisés des tests manuels

Vous avez probablement déjà créé un test sans vous en rendre compte. Vous vous souvenez quand vous avez exécuté votre application et que vous l'avez utilisée pour la première fois ? Avez-vous vérifié le bon fonctionnement des fonctionnalités ? C'est ce qu'on appelle les tests exploratoires et c'est une forme de test manuel.

Les tests exploratoires sont une forme de tests effectués sans plan. Dans un test exploratoire, vous explorez simplement l'application.

Pour disposer d'un test manuel complet, il vous suffit de dresser une liste de toutes les fonctionnalités de votre application, et des différents types d'entrées qu'elle peut accepter et les résultats attendus. Désormais, chaque fois que vous modifiez votre code, vous devez parcourir chaque élément de cette liste et la vérifier.

Ce n'est pas la solution la plus pratique. C'est là qu'interviennent les tests automatisés. Les tests automatisés sont l'exécution de votre plan de test (les parties de votre application que vous souhaitez tester, l'ordre dans lequel vous souhaitez les tester et les réponses attendues) par un script au lieu d'un humain. Python fournit un ensemble d'outils et de bibliothèques pour vous aider à créer des tests automatisés pour votre application.

Différencier les tests unitaires et les tests d'intégration

Le monde des tests ne manque pas de terminologie et maintenant que vous connaissez la différence entre les tests automatisés et manuels, il est temps d'aller plus loin.

Pensez à la façon dont vous pourriez tester les lumières d'une voiture. Vous allumeriez les lumières (comme l'étape de test) puis vous sortez de la voiture ou demandez à un ami de vérifier que les lumières sont allumées (c'est l'étape d'assertion de test). Le test de plusieurs composants est appelé test d'intégration.

Pensez à tout ce qui doit fonctionner correctement pour qu'une tâche simple soit effectuée avec succès. Les composants de la voiture sont comme les composants de votre application, pour nous il s'agit des classes, fonctions et modules.

Le défi majeur avec les tests d'intégration est de poser le diagnostic en cas de résultat erroné. Il est difficile de diagnostiquer le problème sans pouvoir isoler quelle partie du système est en panne. Si les lumières ne s'allument pas, les ampoules sont peut-être cassées ou la batterie est vide.

Un test unitaire est un test plus petit, qui vérifie qu'un seul composant fonctionne. Un test unitaire vous aide à isoler ce qui est cassé dans votre application et à le réparer plus rapidement.

Tests fonctionnels

Les tests fonctionnels se concentrent sur une séquence d'opérations qui ont pour objectif de vérifier le parcours d'un utilisateur et s'assurer qu'une fonctionnalité respecte le comportement attendu. Le logiciel doit renvoyer les informations attendues par l'utilisateur quand il réalise une action spécifique.

Ces tests reprennent souvent un parcours utilisateur. Un test fonctionnel est associé à une User Story et teste son bon déroulement avant l'étape manuelle. Pour effectuer ces tests, vous aurez besoin d'importer la librairie Selenium comme ceci.
