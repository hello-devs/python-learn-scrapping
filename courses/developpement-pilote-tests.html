Développement piloté par les tests

Durée : 1 h

Environnement de travail : Visual Studio Code (ou tout autre éditeur de texte), PHP, PHPUnit

Pré-requis : PHP, Les tests unitaires avec PHPUnit, Les tests fonctionnels avec WebTestCase et Panther, Les test unitaires avec les mocks

Contexte :

Le Test Driven Development (ou développement guidé par les tests) est une technique de développement consistant à écrire les tests avant les méthodes et classes à tester.

Effectivement, ce sont les tests qui guident le développement et non plus l'inverse.

Dans la pratique, il nous faudra connaître tous les comportements découlant d'une fonctionnalité avant d'écrire un test correspondant à un de ces comportements. On s'assure qu'il n'est pas valide, étant donné que le code n'a pas encore été rédigé, puis on écrit ce fameux code pour que le test soit valide avant de passer à une étape de refactorisation et d'optimisation.

Le Test Driven Development (ou TDD) consiste donc à réaliser ces traitements pour chacun des comportements attendus par la fonctionnalité que l'on souhaite implémenter. En suivant cette méthodologie, notre code est testé, écrit et optimisé. L'objectif étant d'offrir une plus grande maintenabilité à notre code source.

Objectifs

  • Comprendre le TDD et ses avantages

  • Découvrir les 5 étapes du TDD

Contexte :

Le Test Driven Development va donc consister à rédiger les tests concernant une fonctionnalité ou un service avant son implémentation.

En réalité le TDD, peut être découpé en 5 étapes à suivre :

  • Rédiger le test,

  • S'assurer que le test n'est pas valide,

  • Rédiger un code suffisant pour valider le test,

  • S'assurer que le test est valide,

  • Reprendre le code ainsi testé pour l'optimiser.

Dans l'idée, il s'agit de tester, rendre fonctionnel et refactoriser.

En suivant ces étapes, dans cet ordre, le TDD vous incite à réfléchir à la manière dont vous allez implémenter votre code en amont.

Il existe trois différents types de test :

  • Les tests unitaires permettent de tester une méthode spécifique d'une classe précise.

  • Les tests fonctionnels permettent d'automatiser des scénarios de tests que l'on commande à un navigateur web.

  • Les tests d'intégration sont, en réalité, des tests unitaires qui ne se basent, cette fois, non pas sur de fausses données mais sur une vraie base de données.

Ayant connaissance de ces différentes pratiques une question vous vient peut-être à l'esprit. Suis-je censé tester toutes mes fonctions à l'aide de tests unitaires et tous mes formulaires ainsi que toutes les validations qui en découlent à l'aide de tests fonctionnels ?

Bien heureusement, non ! Retenez que quelques tests valent toujours mieux qu'aucun.

En réalité, rédiger trop de tests va vous prendre beaucoup de temps, vous ralentir pour, au final, une très faible valeur ajoutée. Non, nous allons plutôt nous contenter de tester les fonctionnalités qui nous paraissent complexes.

Enfin, nous laisserons nos tests guider le développement des différentes fonctionnalités et comportements que nous souhaitons implémenter.

En effet, le TDD est une pratique offrant plusieurs avantages :

  • Tester différents cas de figure,

  • Découper le code en unités,

  • Proposer des optimisations/refectorisations validées avec les mêmes tests,

  • Gagner du temps en validant notre code via la console et non plus via le navigateur.

Vous l'aurez compris, si vous avez l'intention d'écrire un test, favorisez toujours une approche TDD.
