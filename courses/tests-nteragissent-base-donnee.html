Tests qui interagissent avec une base de données

Durée : 1 h

Environnement de travail : Visual Studio Code (ou tout autre éditeur de texte), PHP, Composer, Symfony CLI, PHPUnit

Pré-requis : PHP, les tests unitaires avec PHPUnit, Doctrine

Contexte :

Pour rappel, un test unitaire consiste à tester une unité de code. Il s'agit bien souvent de tester une méthode en particulier. Pour autant, une méthode fait souvent appel à un ou plusieurs objets issus d'autres classes.

Ainsi, si l'on souhaite tester une méthode faisant appel à de tels objets, il nous faut utiliser des mocks. En bref, il s'agit d'objets simulés.

Dans le cadre de nos tests, ces mocks pourront également être utiles pour tester des méthodes faisant appel à une ou plusieurs entités de notre base de données.

Objectifs

  • Créer et configurer une base de données de test

  • Configurer PHPUnit

  • Créer des fixtures de test

Mise en situation

Avant de nous attarder davantage sur la mise en place et l'utilisation des mocks, nous allons découvrir comment tester les méthodes faisant appel à notre base de données.

Pour ce faire, nous créerons une base de données dédiée aux tests, avant de la configurer dans les variables d'environnement de notre projet Symfony.

Nous prendrons également quelques minutes pour configurer correctement le framework de test PHPUnit, en passant par son fichier de configuration XML.

Il ne nous restera plus qu'à créer des fixtures de test. Les données que l'on créera de cette manière sont appelées des dummy data (ou « données factices »).

Dans cette partie, nous allons nous concentrer sur les tests permettant d'interagir avec une base de données. Jusqu'ici, tous les tests unitaires que nous avons rédigés ont été mis en place de manière complètement isolée.

Pour autant, il arrive bien souvent que nos composants ou contrôleurs à tester interagissent avec une base de données. Pour tester ce genre de bouts de code, nous allons avoir besoin d'accéder à une base de données de test.

Voyons comment configurer une telle base de données à l'aide du système de fixtures proposé par Doctrine. Ce système permet de rédiger une méthode chargée d'indiquer à Doctrine comment compléter notre base de données.

Il est accessible en ligne de commande afin de réaliser simplement les différents traitements.

Méthode :

Pour utiliser le système de fixtures, il nous faut d'abord installer la dépendance au sein de notre projet Symfony, en passant par une simple commande Composer : $ composer require --dev orm-fixtures.

Vous l'aurez compris, cette commande installe automatiquement une recette dans notre projet, elle-même chargée d'installer le système de fixtures de Doctrine, et le bundle qui va avec, avant de l'activer.

Ceci étant fait, un nouveau dossier fait son apparition dans le répertoire /src de votre projet Symfony. Il s'agit du répertoire DataFixtures. C'est ce dossier qui contiendra les différentes classes de fixtures que nous voudrons utiliser par la suite.

Dans le cadre de ce chapitre, nous souhaitons créer une fixture permettant d'ajouter 10 produits en base de données.

Pour ce faire, il nous faut d'abord créer une entité Product. Pour faciliter les choses, il est conseillé d'utiliser le bundle Maker. Pour l'installer, il suffit de saisir la commande Composer $ composer require --dev symfony/maker-bundle.

Une fois le bundle correctement installé, vous aurez accès à la commande $ php bin/console make:entity afin de créer très rapidement et simplement une entité Doctrine.

Assurez-vous d'avoir préalablement installé Doctrine au sein de votre projet Symfony : $ composer require symfony/orm-pack.

Nous avons choisi de générer l'entité Product avec deux propriétés : name et price.

Avant de créer notre fixture, nous allons configurer notre base de données de test. Pour ce faire, rendez-vous dans le fichier .env, situé à la racine de votre projet Symfony. Ici, nous allons compléter l'URL pointant vers notre de base de données de test (qui n'existe donc pas encore) grâce à la variable d'environnement DATABASE_URL.

Dans mon cas, j'utilise le système de gestion de base de données MySQL inclus dans le pack MAMP et je souhaite stocker mes données dans une base intitulée « studi_test ». Voici à quoi ressemble désormais ma variable d'environnement DATABASE_URL , située dans le fichier .env.

1
DATABASE_URL="mysql://root:root@127.0.0.1:3306/studi_test?serverVersion=5.7"
DATABASE_URL="mysql://root:root@127.0.0.1:3306/studi_test?serverVersion=5.7"

Il faut ensuite exécuter la commande $ php bin/console doctrine:database:create --env=test pour créer la base de données, tout en précisant avec le drapeau --env qu'il s'agit d'une base de données prévue pour la réalisation de tests uniquement.

Bien entendu, la base de données ainsi créée est vide. Il va falloir l'inciter à reprendre la structure de l'entité Product pour ainsi créer la table associée. Pour ce faire, il suffit de saisir la commande $ php bin/console doctrine:schema:update --env=test --force. Remarquez que l'on utilise toujours le drapeau --env pour préciser l'environnement de test, associé cette fois au drapeau --force pour préciser à Doctrine que l'on souhaite qu'il exécute les requêtes SQL qui s'imposent, sans qu'il ne cherche à les afficher.

Notre base de données de test a désormais la bonne structure et est parfaitement prête à être utilisée.

Étant donné que nos tests sont censés être indépendants les uns des autres, il est essentiel de s'assurer que l'exécution d'un test ne va pas supprimer ni modifier des données, afin de ne pas impacter sur l'exécution d'un autre.

Pour cela, on va installer un paquet garantissant que chaque test est exécuté avec la même base de données non modifiée. Pour installer cette dépendance, il suffit de saisir la commande Composer suivante : $ composer require --dev dama/doctrine-test-bundle avant d'activer l'extension PHPUnit qui en découle directement dans le fichier de configuration du framework. Vous l'aurez compris, il s'agit du fichier phpunit.xml.dist. Pour ce faire, ajoutez simplement ces trois lignes au fichier :

class ProductFixtures extends Fixture {

        for($i = 0; $i < 10; $i++) {
            $product = (new Product())
                ->setName("Sauce tomate$i")
                ->setPrice($i);
            $manager->persist($product);

Pour fonctionner, une classe de fixture doit absolument étendre le bundle Doctrine\Bundle\FixturesBundle\Fixture afin d'implémenter la méthode load() qui sera appelée à l'exécution de nos fixtures.
Comme vous pouvez le constater, cette méthode reçoit nécessairement un objet $manager issu de la classe Doctrine\Persistence\ObjectManager en argument. C'est cet objet que nous allons utiliser pour persister, puis synchroniser nos dummy data en base de données.
Après quoi, on se contente de boucler 10 fois sur quelques instructions se chargeant de créer un nouveau produit, de lui donner un nom et un prix fictif avant de les persister individuellement. On synchronise enfin la base de données avec les objets ainsi créés.
Pour exécuter cette fixture, il suffit de saisir la commande $ php bin/console doctrine:fixtures:load en console :
Après avoir précisé que l'on est bien d'accord avec les changements que cela va impliquer en base de données, on constate que les méthodes load() implémentées dans les classes AppFixtures et ProductFixtures sont exécutées.

Syntaxe : À retenir
Cette partie du cours comporte plusieurs éléments essentiels en termes de syntaxe. La commande $ composer require --dev orm-fixtures va vous permettre d'installer toutes les dépendances nécessaires à l'utilisation du système de fixtures proposé par Doctrine.
Souvenez-vous également qu'il vous faudra passer par une étape de configuration de votre base de données de test à travers la mise à jour du fichier .env, et plus précisément de la variable d'environnement DATABASE_URL, qui devra contenir une chaîne de caractères représentant le lien pointant vers votre base de données de test.
Pour créer cette base de données en environnement de test, il suffit de saisir la commande $ php bin/console doctrine:database:create --env=test avant de mettre à jour sa structure en cohérence avec vos entités Doctrine, à l'aide de la commande $ php bin/console doctrine:schema:update --env=test —force.
Il vous faudra également passer par une étape de configuration de PHPUnit. Pour cela, il vous faut installer puis activer le paquet dama/doctrine-test-bundle via la commande Composer $ composer require --dev dama/doctrine-test-bundle.
Enfin, pour créer une fixture, il nous faut une classe étendant le bundle Doctrine\Bundle\FixturesBundle\Fixture et implémentant la méthode load(), censée recevoir une instance de la classe Doctrine\Persistence\ObjectManager en argument.
Une fois la fixture correctement implémentée, la méthode load() est exécutée au lancement de la commande $ php bin/console doctrine:fixtures:load.
