Documenter pour maintenir et faire évoluer

Contexte :

Bienvenue dans ce cours destiné à vous expliquer comment fonctionnent les recettes et quels sont les outils à mettre en place pour traiter rapidement les anomalies rencontrées à la conception de votre solution ou durant sa maintenance. Nous évoquerons les différentes problématiques liées à la Création de documentation pour faire évoluer la solution, nous verrons également quelle solution utiliser pour répertorier les différentes anomalies sur un code et surtout comment et quand sont-elles corrigées, pour finir, nous verrons quelques éléments pour mesurer les performances et l'activité de sa solution.

Introduction

Lorsque vous mettez en place une solution, cette conception se passe en deux étapes généralement.

  • Dans un premier temps, vous mettez à plat toutes les fonctionnalités que vous souhaitez intégrer dans votre solution.

  • Cependant, au moment du développement, il arrive souvent que les fonctionnalités ne peuvent pas toujours être développées à cause de contraintes de temps ou financières.

    • Il est donc important de garder ces idées sur un document pour faire évoluer la solution avec une v2, une v3 etc.

Il est donc important de mettre en place différentes documentations pour maintenir et faire évoluer notre solution.

Pourquoi documenter ?

La documentation est, contrairement à ce qu'on pourrait penser, un des éléments les plus importants dans la conception d'une solution.

Bien qu'elle soit parfois compliquée à mettre en place, elle est nécessaire à toute personne ne connaissant pas la solution de l'utiliser ou alors même développer dessus si le développeur en question n'a jamais touché au code de la solution.

Quelles sont les méthodes de maintenances et d'évolution

La mise en place d'une maintenance peut se faire de différentes méthodes, chaque méthode possède ses avantages et ses inconvénients mais s'adapte plus ou moins facilement à votre solution et à l'approche que vous souhaitez mettre en place sur votre documentation.

Il existe donc plusieurs méthodes de documentation pour assurer la maintenance et l'évolution d'une solution, nous en verrons 2 pas forcément orientées digitales que nous explorerons dans le détail car elles apportent un regard assez global.

Une fois que nous comprenons ces méthodes, il sera facile de mettre en place les documentations associées pour expliquer les processus à suivre.

  • La TPM

    • Note : la TPM est principalement orientée industrielle, mais il est parfaitement possible de l'adapter à des solutions digitales.

    • La TPM pour Total productive maintenance ou Maintenance productive totale en français est une méthode japonaise. Elle permet de mettre en place une évolution des différentes méthodes de maintenance pour augmenter le rendement des services. Son objectif premier est d'éviter un maximum l'arrêt inopiné des services, la TPM permet aussi d'optimiser le temps d'intervention des techniciens.

    • Cette méthode de maintenance va se baser sur trois outils distincts

      • Le TRS (Taux de Rendement Synthétique) :

        • Le TRS est un indicateur permettant de mesurer le taux d'utilisation de la solution par rapport à son taux de rendement maximal. Le TRS est un élément fondamental pour déterminer si la solution tourne à plein régime ou si elle bénéficie encore de performances pour être encore plus efficiente.

      • Les 5s :

        • Les 5S sont 5 mots japonais permettant d'indiquer comment mettre en place une méthode de maintenance.

          • Seiri : ordonner

            • Seiri signifie ordonner sa solution, bien organiser son code et surtout les éléments qui constituent son environnement.

          • Seiton : ranger

            • Seiton signifie le rangement. Dans le cas d'une solution digitale, il est important de ne pas laisser de fichiers volants, si tel est le cas il est possible que ce soit vecteur de dysfonctionnement. Le fichier ne peut être récupéré dans la mauvaise arborescence et cela provoque un crash. Il est donc important de ranger toute l'arborescence de sa solution comme indiqué dans la documentation de l'architecture.

          • Seiso : dépoussiérer

            • Seiso est un point plus orienté industriel, mais dans notre cas digital, Seiso signifie surtout de nettoyer son code pour supprimer les lignes inutiles ou obsolètes. L'objectif d'un « dépoussiérage » permet d'optimiser le fonctionnement de l'appareil ou dans notre cas, de la solution

          • Seiketsu : maintenir l'ordre

            • Une fois que vous avez mis en place cette nouvelle maintenance, il est important de la faire perdurer le plus longtemps possible pour éviter une nouvelle intervention.

          • Shitsuke : rigueur

            • Vous appliquez les 4S précédents, il est donc important de toujours les appliquer dans le bon ordre, avec rigueur pour s'assurer d'une maintenance optimale.

      • L'auto maintenance :

        • Votre solution est optimale mais il est intéressant de proposer une fonction permettant de réaliser de manière automatique des petites actions de maintenance permettant de faire perdurer le bon fonctionnement de votre solution.

Comment mettre en place une documentation technique pour la maintenance évolutive de votre solution en appliquant la méthode TPM ?

Méthode :

Pour mettre en place votre documentation, il est nécessaire dans un premier temps qu'elle reprenne tous les points fondamentaux de votre méthode de travail, vous devrez respecter un sommaire reprenant cette structure :

  1. Présentation de la situation actuelle avec les dysfonctionnements rencontrés

    • 1.1°/ Présentation de la méthode de maintenance utilisée (TPM).

  2. Indication du TRS

    • 2.1°/ Indication du TRS en calculant : indicateur réel/production maximale théorique → Le TRS se calcule en pourcentage. S'il est à 100 %, cela signifie qu'il est optimal. En revanche, s'il est à 0, la solution n'est pas du tout efficace. Les métriques de calculs seront sur les temps de traitement réels par les temps de traitement optimaux.

      Exemple : temps de réponse réel : 20 ms temps de réponse optimal : 17 ms.

      TRS = 17/20 = 0,85 = 85 %.

LE PDCA

Le PDCA est connu plus communément sous le nom d'amélioration continue. Elle permet de mettre en place un cercle vertueux à travers 4 étapes :

  • P : Plan

    • Le Plan permet de mettre en place une planification des éléments à maintenir ou à faire évoluer.

  • D : Do

    • Une fois que le plan a été mis en place, il faut maintenant l'appliquer, une fois appliqué.

  • C : Check

    • Il est important de vérifier les éléments travaillés et s'assurer de la conformité avec les éléments prévus à l'origine lors de la planification des éléments à mettre à jour.

  • A : Act

    • Une fois que la vérification est faite, il est très fréquent qu'un petit pourcentage ne réponde pas totalement à la demande initiale, il est donc impératif de réagir et de régler les anomalies puis pour finir, rédiger la documentation reprenant les différents éléments de maintenance.

Méthode : Comment documenter une maintenance PDCA ?

Documenter une maintenance PDCA est assez facile en soi quand nous comprenons le processus, cependant il est important de se rappeler que la PDCA est constante, donc il est nécessaire de versionner le document.

Chaque tour de maintenance doit se matérialiser par une nouvelle version du process en indiquant les 4 points du process :

  1. La planification des tâches de maintenance

    • Qui peut être matérialisée par un rapport de bugs sur la solution.

  2. Les actions mises en place pour corriger les anomalies

    • Mettre les différents fichiers modifiés ainsi que le code ajouté ou modifié.

  3. Indiquer les retours de modification

    • Auditer si chaque bug évoqué à été corrigé.

  4. Si nécessaire, réagir aux commentaires évoqués

Une fois le tour fait, la nouvelle version de la documentation est prête, tout en bénéficiant de manière incrémentale, toutes les autres maintenances faites au préalable.
