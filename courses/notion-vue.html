Notion de vue

Contexte :

Les vues de Django sont des brookers d'informations d'une application Django. Une vue extrait les données de votre base de données (ou d'une source de données ou d'un service externe) et les fournit à un modèle. Pour une application web, la vue fournit le contenu et les modèles de pages web ; pour une API RESTful, ce contenu peut par exemple être des données JSON formatées.

Lors de la création de notre site web, startproject crée un fichier urls.py dans notre dossier \myclub_site\urls.py. C'est un bon endroit pour la navigation à l'échelle du site, mais c'est rarement un bon endroit pour mettre des URLs relatives à des applications individuelles. Non seulement le fait d'avoir toutes nos URLs dans un seul fichier est plus complexe et moins portable, mais cela peut conduire à un comportement étrange si deux applications utilisent une vue avec le même nom.

Pour mieux comprendre le fonctionnement de Django, détaillons ce qu'il vient de se passer :

  1. Notre navigateur a envoyé un message au serveur de développement Django, lui demandant de renvoyer le contenu situé à l'URL racine (http://127.0.0.1:8000/).

  2. Django a ensuite recherché un modèle d'URL correspondant à la requête, en recherchant d'abord au niveau du site urls.py, puis dans chacune des applications, pour un fichier urls.py contenant un modèle qui correspond.

  3. Django vérifie le premier modèle (admin/) dans notre niveau de site urls.py, qui ne correspond pas et passe à la deuxième ligne dans laquelle la chaîne vide (URL racine) correspond.

  4. Le modèle de correspondance inclut le fichier urls.py de l'application events. Fondamentalement, cela indique : « Allez chercher dans l'application events un modèle qui correspond ».

  5. Une fois au niveau de l'application urls.py, la chaîne vide correspond à nouveau. Mais, cette fois, la requête est envoyée à l'index vue.

  6. L'index vue rend ensuite notre message HTML simple à un HttpResponse et l'envoie au navigateur.

  7. Le navigateur rend la réponse et nous voyons notre en-tête de page.

Remarque :

Chaque application Django suit ce même processus de base à chaque fois qu'elle reçoit une requête du navigateur.

Le contenu dynamique

Pour un site interactif moderne, vous souhaitez que le contenu de vos pages change de manière dynamique, en fonction de l'interaction avec l'utilisateur et de la logique du programme.

Toutes les pages n'ont pas besoin d'être dynamiques. Par exemple, les conditions générales et les déclarations de confidentialité ne changent pas régulièrement et sont souvent enregistrées sous forme de pages HTML statiques.
